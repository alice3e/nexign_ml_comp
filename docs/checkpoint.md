1. Описание архитектуры сервиса
Нужно описать общую архитектуру решения и поток данных.
Обязательно:
Основные компоненты системы
(например: UI / API / ML-модель / препроцессинг / постпроцессинг / хранилища).
Как данные проходят по системе:
вход (что приходит на вход),
обработка,
выход (что возвращает сервис).
Где именно используется ML/LLM/мультимодальная модель.
Как сервис будет разворачиваться (локально, в Docker, CPU/GPU).
Формат:
простая схема (блоки и стрелки) или
текстовое описание с перечислением компонентов.
Важно:
Не требуется сложная enterprise-диаграмма — достаточно логически корректной и понятной схемы.
2. Технологический стек
Нужно перечислить конкретные технологии, которые команда планирует использовать, и кратко объяснить выбор.
Минимальный набор:
Язык программирования
Backend-фреймворк / API
ML / CV / LLM библиотеки и модели
Формат взаимодействия (REST, JSON и т.д.)
Инфраструктура:
Docker
CPU / GPU (что именно планируется)
Дополнительно (если есть):
UI
базы данных
очереди
inference-серверы
Для каждого пункта:
что именно используется;
почему выбран именно этот инструмент (1–2 предложения).
3. План реализации (roadmap)
Нужно показать, как команда планирует дойти до работающего прототипа.
Формат: этапы + результат каждого этапа.
Пример структуры:
Анализ задачи и данных
→ понимание входа/выхода, ограничений.
Выбор и первичная проверка модели
→ модель запускается на примере.
Реализация базового пайплайна
→ вход → модель → выход.
Оборачивание в API
→ REST endpoint.
Docker-сборка
→ сервис запускается одной командой.
(Опционально) улучшения
→ ускорение, обратная задача, UI и т.д.
