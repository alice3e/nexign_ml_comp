# Процесс управления инцидентами в IT-поддержке

Создай BPMN диаграмму процесса обработки IT-инцидентов по методологии ITIL.

## Основные участники:
- Пользователь
- Service Desk (1-я линия поддержки)
- Специалисты 2-й линии поддержки
- Специалисты 3-й линии поддержки (эксперты)

## Описание процесса:
1. Пользователь сообщает об инциденте (по телефону, email или через портал)
2. Service Desk регистрирует инцидент в системе и присваивает номер
3. Service Desk определяет приоритет инцидента:
   - Критический (P1) - влияет на бизнес-процессы
   - Высокий (P2) - влияет на работу отдела
   - Средний (P3) - влияет на одного пользователя
   - Низкий (P4) - незначительное неудобство
4. Service Desk пытается решить инцидент самостоятельно (проверка базы знаний)
5. Если инцидент решен на 1-й линии:
   - Уведомление пользователя
   - Закрытие инцидента
   - Запрос обратной связи
6. Если инцидент не решен на 1-й линии:
   - Эскалация на 2-ю линию поддержки
   - Специалист 2-й линии анализирует проблему
7. Если специалист 2-й линии решает инцидент:
   - Обновление базы знаний
   - Уведомление пользователя
   - Закрытие инцидента
8. Если требуется эскалация на 3-ю линию:
   - Привлечение экспертов или вендора
   - Глубокий анализ и решение проблемы
9. Параллельно с решением инцидента:
   - Отправка уведомлений пользователю о статусе
   - Контроль SLA (время реакции и решения)
10. Если SLA нарушен - автоматическая эскалация руководителю
11. После решения инцидента:
    - Документирование решения
    - Обновление базы знаний
    - Закрытие инцидента
    - Опрос удовлетворенности пользователя

## Требования:
- Используй 1 пул с 4 лейнами (Пользователь, Service Desk, 2-я линия, 3-я линия)
- Используй XOR Gateway для определения приоритета и маршрутизации
- Используй подпроцесс для процесса эскалации
- Добавь Timer Boundary Event для контроля SLA
- Используй Message Event для уведомлений пользователя
- Добавь Event-Based Gateway для ожидания ответа от пользователя или истечения времени
